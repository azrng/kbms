## 项目文档生成

### 方案一：仅生成 README.md

```
你的角色：资深技术文档工程师。

核心任务：根据提供的项目信息，生成一份专业、标准、结构清晰的 README.md 文档。

请提供以下项目信息：
•   项目名称：
•   项目简介：(用一句话说明项目的核心功能或价值)
•   技术栈：(例如：后端-AspNetCore 8, 前端-Vue3, 数据库-MySQL/Redis)
•   运行方式：(例如：Docker Compose 启动 或 本地 npm run dev)
输出要求：
1.  结构：严格按以下 Markdown 章节生成，章节顺序不可变更。
2.  风格：语言专业、简洁，面向开发者。
3.  规范：
    ◦   代码块必须注明语言（如bash）。
    *   涉及 SDD（系统设计文档）时，链接路径必须为 ./docs/sdd/。
    *   必须在文档中通过"> 文档版本：v1.0"等形式明确 SDD 版本号。
    *   对不确定的信息，使用 [待补充] 占位。

请开始生成 README.md 内容：

[项目名称]
[一句简短有力的项目 Slogan 或价值主张]
[可选：在此添加版本、构建状态等徽章，例如 ![License](https://img.shields.io/badge/license-MIT-blue.svg)]

📖 项目简介

[在此段落展开描述项目简介，说明其要解决的核心问题、目标用户及主要特点。]

✨ 特性

*   特性一：例如，高性能、模块化设计。
*   特性二：例如，完整的 API 文档。
*   特性三：例如，支持容器化部署。

🛠️ 技术栈

*   后端：[技术名称与关键版本，例如 .NET 8]
*   前端：[技术名称与关键版本，例如 Vue 3 + Vite]
*   数据库：[例如 MySQL 8, Redis 7]
*   部署/运维：[例如 Docker, Nginx]

🚀 快速开始

先决条件

确保你的开发环境满足以下要求：
*   [例如：.NET SDK 8.0+]
*   [例如：Node.js 18+]
*   [例如：Docker & Docker Compose]

安装与运行

1.  克隆项目
    git clone [项目仓库地址]
    cd [项目目录]

2.  配置环境
    cp .env.example .env
    # 请根据实际情况编辑 .env 文件中的配置

3.  启动服务
    # 使用 Docker Compose 启动（示例）
    docker-compose up -d

4.  访问应用
    服务启动后，可通过 http://localhost:[端口号] 访问。
    *   默认测试账号：[如有，请填写，否则用[待补充]]
```

### 方案二：仅生成 SDD 文档

```
# Role
你是一位资深软件架构师，精通 AGI SDD（规范驱动开发）方法论。

# Task
请分析我现有的项目代码，逆向生成一套 SDD 标准文档，并将当前状态固化为 **v1.0 版本**。

# Context
- 项目类型：[例如：电商后台管理系统 / 用户登录服务 / 数据报表平台]
- 技术栈：[例如：AspNetCore + Vue3 + MySQL + Redis]
- 核心功能：[例如：用户管理、订单处理、权限控制]

# Output Requirements
请严格按照 SDD 三大阶段生成以下三个文档内容，并标记版本号为 v1.0：

## 1️⃣ requirements.md (需求阶段)
- **用户故事**：按 "作为 [角色]，我想要 [功能]，以便 [价值]" 格式
- **验收标准**：必须采用 **EARS 格式** (WHEN...THEN...)
  - 示例：WHEN 用户输入错误密码 3 次 THEN 系统锁定账号 10 分钟
- **版本标记**：文档头部注明 `Version: v1.0`

## 2️⃣ design.md (设计阶段)
- **架构流程**：描述请求链路 (如：前端→网关→服务→数据库)
- **数据模型**：核心表结构及字段说明
- **技术选型**：关键组件及选型理由 (如：Redis 存储验证码)
- **版本标记**：文档头部注明 `Version: v1.0`

## 3️⃣ tasks.md (任务阶段)
- **任务清单**：将功能拆解为可独立验证的小任务
- **状态标记**：当前已完成的任务标记为 `[x]`，未完成标记为 `[ ]`
- **版本标记**：文档头部注明 `Version: v1.0`

# Constraints
1. 文档必须与现有代码逻辑保持一致，不能凭空捏造
2. 如发现代码与潜在需求不符，请在文档中标注 `⚠️ 待确认`
3. 输出格式为 Markdown，便于后续版本控制
4. 保持文档"活"性，为后续迭代预留更新空间

# Workflow
1. 先分析我提供的项目代码/结构
2. 生成 v1.0 版本的三份文档草稿
3. 等待我确认后再最终定稿
```

### 方案三：README + SDD 一体化生成

```
# Role
你是一位资深软件架构师，精通 AGI SDD（规范驱动开发）方法论与 EARS 需求语法。

# Task
请分析我现有的项目代码/结构，逆向生成一套完整的 SDD 标准文档体系，并将当前状态固化为 **v1.0 版本**。
同时生成一份项目 README.md 作为文档索引。

# Context
- 项目类型：[例如：电商后台管理系统]
- 技术栈：[例如：AspNetCore + Vue3 + MySQL + Redis]
- 核心功能：[例如：用户管理、订单处理、权限控制]
- 代码结构/关键逻辑：[在此处简要粘贴项目结构树或核心代码片段，若无则让 AI 基于通用最佳实践推断]

# Output Requirements
请严格按照以下 **4 个文件** 生成内容，每个文件使用独立的代码块，并在代码块上方注明 **建议保存的文件路径**。

## 1️⃣ 文件：README.md (项目入口)
- **项目简介**：一句话描述项目价值
- **SDD 文档导航**：提供指向 docs/requirements.md, docs/design.md, docs/tasks.md 的链接
- **快速开始**：简要的安装/运行指引
- **版本信息**：注明当前 SDD 版本为 v1.0

## 2️⃣ 文件：docs/requirements.md (需求阶段)
- **版本标记**：头部注明 `Version: v1.0`
- **用户故事**：按 "作为 [角色]，我想要 [功能]，以便 [价值]" 格式
- **验收标准**：必须采用 **EARS 格式** (WHEN...THEN...)
  - 示例：WHEN 用户输入错误密码 3 次 THEN 系统锁定账号 10 分钟
- **一致性检查**：若发现代码逻辑与潜在需求不符，标注 `⚠️ 待确认`

## 3️⃣ 文件：docs/design.md (设计阶段)
- **版本标记**：头部注明 `Version: v1.0`
- **架构流程**：描述请求链路 (如：前端→网关→服务→数据库)
- **数据模型**：核心表结构及字段说明
- **技术选型**：关键组件及选型理由 (如：Redis 存储验证码)

## 4️⃣ 文件：docs/tasks.md (任务阶段)
- **版本标记**：头部注明 `Version: v1.0`
- **任务清单**：将功能拆解为可独立验证的小任务
- **状态标记**：当前已完成的任务标记为 `[x]`，未完成标记为 `[ ]`
- **迭代预留**：预留未来扩展任务的空间

# Constraints
1. **文档一致性**：文档内容必须与现有代码逻辑保持一致，不能凭空捏造。
2. **EARS 规范**：验收标准必须严格遵循 WHEN...THEN... 结构，确保可测试性。
3. **版本锚定**：所有文档头部必须明确 `Version: v1.0`，体现 Spec-anchored 理念。
4. **格式规范**：输出格式为 Markdown，便于后续版本控制。
5. **文件结构**：明确指示用户创建 `/docs` 文件夹存放 SDD 文档。

# Workflow
1. 分析我提供的项目信息。
2. 一次性生成上述 4 个文件的完整内容。
3. 等待我确认后再进行后续迭代。
```

### 后续迭代建议

| 场景         | 操作建议                                                     |
| ------------ | ------------------------------------------------------------ |
| **新增功能** | 先更新 `requirements.md` → 再更新 `design.md` → 最后执行代码 |
| **修复 Bug** | 在 `tasks.md` 中新增修复任务，并更新验收标准                 |
| **架构调整** | 升级文档版本至 v2.0，保留 v1.0 作为历史参考                  |
| **团队协作** | 将 SDD 文档纳入 Code Review 流程，确保"说的"和"做的"一致     |

## 架构图 HTML 生成

```
你是资深软件架构师和前端可视化设计师。请基于当前项目代码，生成一个「对外说明 + 工程概览」视角的静态 HTML 架构图，风格参考企业级产品架构图、云厂商解决方案架构图和技术汇报架构海报。

不要生成 Markdown，不要生成 Mermaid，不要生成图片说明，直接输出完整可保存运行的 HTML 文件内容。

目标产物：
- 文件名建议：doc/{项目名}-2-architecture.html
- 单文件 HTML，包含内联 CSS
- 不依赖外部图片、字体、JavaScript、CDN 或第三方库
- 打开 HTML 即可查看
- 页面语言使用中文
- 适合放进项目 doc 目录作为架构说明材料
- 页面应更接近「一张横向架构图海报」，不是长篇说明页

视觉参考方向：
- 最终页面必须像「一张可用于汇报的架构图」，而不是普通文档网页。
- 优先使用 16:9 或接近 1600x1000 的宽屏画布，核心信息尽量首屏完整呈现。
- 中央主流程必须是视觉重心，其他模块围绕主流程排布。
- 参考企业技术方案图：左侧触发源，中间 Pipeline，右侧外部系统，下方核心模块、数据流、横切能力和图例。
- 信息要高密度但清晰，适合截图放进技术方案、汇报材料或项目文档。
- 标题本身不参与编号；画布中的主要可见区域从「1. 触发与入口」开始连续编号。
- 区域编号必须按实际展示区域连续重编号，不能出现跳号。如果某个区域不存在或被合并，后续区域自动前移编号。
- 推荐区域顺序为：触发与入口、主业务 Pipeline、外部依赖与集成、核心数据模型、核心模块、数据流向、横切能力、输出与应用 / 图例；实际输出可以少于 8 个区域，但编号必须连续。
- 每个能力、输出或模型项除标题外，还要有一条 4 到 10 个字左右的子描述，例如「重试机制 → 指数退避」「缓存控制 → 容量上限」。

分析要求：
1. 先阅读项目结构、入口文件、主要模块、服务层、数据访问层、配置、测试和文档。
2. 识别项目的真实技术栈，例如 UI 框架、后端框架、数据库、Agent / AI Runtime、第三方渠道、持久化方式、测试工程等。
3. 不要编造不存在的模块、外部系统、数据库表、接口、消息队列、权限系统或 AI 能力。
4. 如果某项能力只是代码中有雏形、配置入口或预留扩展点，要用「可选」「预留」「当前支持」「当前未接入」等措辞区分，不要写成已完整上线。
5. 架构图要面向外部读者理解系统，不要只画源码目录层次。源码层次图可以作为辅助信息，但不要和整体产品架构图混成一张目录树。
6. 对所有组件名、类名、目录名、配置名、数据模型和外部依赖，必须从项目真实代码、配置或文档中提取。

页面内容结构：
标题：
   - `{项目名} 整体架构`
   - 标题居中展示，不参与区域编号

1. 触发与入口：
   - 用户入口
   - 外部渠道入口
   - 定时任务 / 恢复任务 / 配置入口，如项目存在则展示
   - 如果不存在某类入口，不要强行补齐；可以省略或标注「当前未发现」

2. 主业务 Pipeline：
   - 用 6 到 8 个横向步骤展示核心生命周期
   - 每步包含：序号、短标题、核心组件名、简短说明
   - 每个步骤都要配一个简洁图标或 CSS 符号，例如齿轮、刷新、下载、代码、数据库、对勾、HTTP、API 等
   - 步骤之间用 CSS 箭头连接
   - 需要体现项目真实存在的失败恢复、重试、审批、缓存、压缩、限流、幂等、异常处理、事务或迁移机制
   - 如果项目没有这些机制，不要编造；只展示真实存在的机制

3. 外部依赖与集成：
   - 先判断项目是否真实存在 AI、Agent、模型服务或智能 Runtime
   - 如果项目存在 AI 能力，展示模型服务、Agent Runtime、工具调用、上下文处理、审批或安全边界
   - 如果项目没有 AI，不要强行出现 AI 专区；侧栏标题直接改成「外部依赖与集成」或「外部接口与渠道」
   - 无 AI 项目只需在外部依赖侧栏或事实说明中简短标注「当前未接入 AI / Agent Runtime」，不要反复强调
   - 展示第三方 API、渠道网关、消息平台、远程服务、文件系统、更新源等真实外部依赖

4. 核心数据模型：
   - 列出项目中最重要的实体、DTO、配置对象、持久化对象或任务模型
   - 每项一句话说明用途
   - 如果存在任务模型、运行状态模型、报表任务模型或核心聚合根，优先放在右侧高面板，和主业务 Pipeline 形成并列关系
   - 如果模型项较多，可在右侧高面板中使用紧凑列表或 2 列小卡片，但不要让它挤压主流程
   - 不要虚构数据库表、字段或 DTO

5. 核心模块：
   - 按真实分层展示，例如：
     - View / UI
     - ViewModel / Application
     - Service
     - Agent / Tool / Parser
     - Repository / Storage
     - Channel / Integration
   - 每个模块列出 3 到 6 个关键类、目录或职责
   - 每个模块卡片都要配一个简洁图标或 CSS 符号，例如 UI、VM、Service、Parser、DB、Channel 等
   - 模块命名必须来自真实项目结构或代码职责

6. 数据流向：
   - 从用户输入 / 外部请求 / 定时触发开始
   - 经过应用编排、上下文 / 规则处理、业务服务、工具 / 外部调用、持久化
   - 最终输出到 UI、API、消息渠道、文件、数据库、报表或发布产物
   - 用横向流程卡片展示，不要写成纯段落
   - 数据流每个阶段要使用差异化配色，按职责区分输入、解析、标准化、存储、消费或输出
   - 每个数据流阶段都要包含阶段名称、真实组件或数据形态、简短子描述

7. 横切能力：
   - 配置管理、权限、日志、审计、测试、错误处理、缓存、上下文压缩、发布部署、迁移恢复、幂等、限流等
   - 只展示项目真实存在或明显由代码支持的能力
   - 每个横切能力项必须包含「能力标题 + 简短子描述」，例如「重试机制 / 指数退避」「限流控制 / API 调用限流」「迁移恢复 / 旧数据归并」
   - 每个横切能力项都要配一个简洁图标或 CSS 符号，例如循环、仪表、盾牌、日志、播放、齿轮等
   - 如果项目没有某项能力，不要为了凑数添加；优先减少卡片数量

8. 输出与应用 / 图例：
   - 展示最终交付给用户或外部系统的结果形态
   - 例如：页面视图、数据文件、报表、通知、接口响应、导出包、发布包、任务状态等
   - 每个输出项都要配一个简洁图标或 CSS 符号，例如图表、铃铛、下载、API、省略号等
   - 图标只用于辅助识别，不要使用外链图片
   - 图例展示主流程数据流、重试 / 恢复 / 反馈流、读写交互
   - 图例样式必须和页面中的箭头样式一致

布局约束：
- 不要把所有内容按网页文章纵向堆叠。
- 主流程区域占页面上半部分最大宽度，步骤卡片横向串联。
- 左侧放「触发与入口」，中间放「主业务 Pipeline」，右侧放「外部依赖 / API / 第三方系统」和「核心任务 / 数据模型」。
- 数据模型优先放在右侧，与 Pipeline 或外部依赖形成垂直高面板；不要默认放到页面底部。
- 中部放「核心模块」，可横向排列 4 到 6 个模块卡片。
- 底部放「数据流向」「输出与应用」「横切能力」和「图例」。
- 如果某个推荐区域不存在，允许省略或合并到相邻区域；省略后必须重新连续编号，并在内容上保持布局紧凑。
- 页面整体高度不要无限拉长，优先做成可截图、可汇报的一张图。
- 如果信息过多，优先删减说明文字，不要缩小字体到难读，也不要让页面变成很长的文档。
- 宽屏布局建议页面宽度 1400px 到 1600px。
- 页面最小宽度可设置为 1120px 或更高，优先保证架构图不乱。

建议布局示意：

    下面编号是推荐区域顺序。实际项目缺少某个区域时，删除该区域并重新连续编号。

    ┌─────────┬────────────────────────────┬─────────────────┐
    │ 1. 触发 │ 2. 主业务 Pipeline          │ 3. 外部依赖     │
    │ 与入口  │ 6 到 8 步横向主流程          │ 4. 数据模型     │
    ├─────────┴────────────────────────────┴─────────────────┤
    │ 5. 核心模块：4 到 6 个真实分层模块横向网格               │
    ├───────────────────────────────────────────────────────┤
    │ 6. 数据流向：输入 → 解析/处理 → 标准化 → 存储 → 输出      │
    ├──────────────────────────────┬────────────────────────┤
    │ 7. 横切能力：能力 + 子描述     │ 8. 输出与应用 + 图例     │
    └──────────────────────────────┴────────────────────────┘

图形表达要求：
- 每个流程步骤必须是独立卡片，卡片之间使用 CSS 箭头连接。
```
